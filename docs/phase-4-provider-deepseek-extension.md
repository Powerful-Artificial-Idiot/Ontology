## Provider Boundary

```text
LlmSemanticParser / LlmEvidenceAnswerComposer
  -> StructuredSemanticProvider / StructuredAnswerProvider
  -> StructuredOutputProvider
       |-- OpenAI Responses API: strict-json-schema
       `-- DeepSeek Chat Completions: json-object
  -> existing local runtime validators
  -> canonical reconstruction / evidence and citation gates
```

The Pipeline does not know the selected provider, HTTP transport, endpoint, model, or credential. DeepSeek Semantic and Answer adapters share one transport client while retaining their separate governed schemas and local validators.

DeepSeek uses its native API contract:

- base URL: `https://api.deepseek.com`
- endpoint: `POST /chat/completions`
- authentication: server-side Bearer token
- supported models: `deepseek-v4-flash`, `deepseek-v4-pro`
- response mode: `{ "type": "json_object" }`
- thinking: explicitly disabled
- tools: not supplied

The client consumes only `choices[0].message.content`. It does not consume or persist `reasoning_content`, prompts, raw responses, or chain-of-thought. JSON Object mode is not treated as server-side JSON Schema enforcement; the existing strict local semantic and answer validators remain authoritative.

Official API references:

- [DeepSeek Chat Completion](https://api-docs.deepseek.com/api/create-chat-completion)
- [DeepSeek JSON Output](https://api-docs.deepseek.com/guides/json_mode/)
- [DeepSeek Models](https://api-docs.deepseek.com/api/list-models/)

## Configuration

Deterministic and template modes remain the defaults. Select DeepSeek only in the server process:

```bash
export MKG_LLM_PROVIDER=deepseek
export MKG_DEEPSEEK_API_KEY=<server-secret>
export MKG_DEEPSEEK_MODEL=deepseek-v4-flash
export MKG_DEEPSEEK_ANSWER_MODEL=deepseek-v4-flash
export MKG_AGENT_SEMANTIC_PARSER_MODE=llm
export MKG_AGENT_ANSWER_COMPOSER_MODE=llm
```

`MKG_DEEPSEEK_BASE_URL` defaults to the official API root. Model selection is configuration, not business logic. Unknown or retired model IDs fail startup validation instead of silently changing behavior.

## Acceptance

Run the DeepSeek live acceptance:

```bash
npm run deepseek:acceptance
```

It writes `.data/evaluations/deepseek-provider-acceptance.json` with independent statuses for Semantic Parser, Answer Composer, and the complete governed pipeline. The full pipeline assertion requires canonical OP30 and Leak Rate resolution, the approved read-only query template, five governed evidence items, and a passing deterministic citation gate.

Every artifact generated by the current provider acceptance runner records `fallbackUsed: false`; older `1.0.0` and `1.1.0` artifacts remain readable for backward compatibility. The `1.2.0` artifact records Quality, Engineering Change, Bottleneck, and cross-domain acceptance independently. The live runner has no Mock, deterministic parser, or template-answer availability fallback; a provider error produces `failed` rather than a passing live status.

Run OpenAI acceptance independently:

```bash
npm run openai:acceptance
```

Without a real key, commands write a transparent `pending` artifact and exit without making a provider call. Provider failures produce `failed`; there is no silent Mock or deterministic success fallback in live acceptance.

## Security And Governance

- Credentials remain server-only and are never included in Vite variables.
- Provider telemetry contains provider/model/transport/status/latency/token metadata only.
- HTTP error bodies and raw model output are not retained.
- DeepSeek cannot generate or execute Cypher, access Neo4j, create canonical IDs or Evidence Packs, validate citations, or decide publication.
- Cancellation and timeout continue through the shared `AbortSignal` and typed Agent error boundary.
